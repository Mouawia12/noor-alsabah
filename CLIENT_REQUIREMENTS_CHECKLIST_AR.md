## 2) القرارات المحسومة + Stack

> **قرارات مثبّتة مع العميل/المنفّذ:**
> 1. **محرك الاستخراج:** OpenAI (رؤية) — مع طبقة خدمات معزولة قابلة لتبديل المحرك.
> 2. **حدود الفاتورة:** كشف ذكي بالـAI (يدعم فاتورة-بصفحة / متعددة الصفحات / عدة فواتير بصفحة).
> 3. **اللغات:** مختلط عربي/إنجليزي.
> 4. **قنوات التنبيه:** داخل النظام + البريد الإلكتروني.

| الحاجة | الأداة |
|--------|--------|
| استخراج بيانات الفواتير/العقود (رؤية + JSON منظّم) | **OpenAI Vision** — `gpt-4o-mini` للحجم الكبير اقتصادياً + تصعيد `gpt-4o` عند انخفاض الثقة، عبر **Structured Outputs (json_schema)** لضمان مخرجات منظّمة موثوقة |
| كشف حدود الفواتير | تصنيف "بداية فاتورة جديدة؟" بالـAI + heuristics (تغيّر الرقم/المورد) |
| تقسيم/استيراد صفحات PDF | `setasign/fpdi` (+ tcpdf الموجود) |
| تحويل صفحة PDF ممسوحة إلى صورة | Imagick + Ghostscript (تبعية سيرفر) |
| قراءة نص PDF النصي | `smalot/pdfparser` |
| المعالجة الخلفية للدفعات الكبيرة | Laravel Queue (`database` أو Redis) + Jobs |
| عميل HTTP | `guzzlehttp/guzzle` (موجود) |
| التنبيهات | إشعارات Laravel (قناة database داخل النظام + قناة mail) |
| تصدير التقارير | dompdf + phpspreadsheet (موجودان) |

## 7) المرحلة 9 — تحسينات ما بعد التسليم (ملاحظات العميل 29–30 يونيو 2026)

> مصدر هذه النقاط: تجربة العميل الفعلية على الخادم + مطالب رسمية مرسلة. الأولوية القصوى = **قوة ودقّة استخراج الفواتير** وفرزها وترحيلها.

### أ) قوّة ودقّة استخراج الفواتير — 🔴 الأولوية القصوى
> العَرَض الذي أبلغ عنه العميل: «رفعت 10 فواتير ولم يُقبل إلا 3، ولم أعرف ما الفواتير المرفوضة».
- [ ] **إصلاح الدمج الخاطئ (`PageMergeService`)** الذي يطوي فواتير حقيقية في الفاتورة السابقة عندما يفشل الاستخراج في قراءة رقم الفاتورة + الإجمالي معاً → هذا يُخفي فواتير ويُنقص العدد بصمت (السبب الأرجح لاختفاء 7 فواتير).
- [ ] رفع جودة الاستخراج: نموذج أقوى كأساس و/أو محرك OCR متخصّص للفواتير (انظر القرار «ز»).
- [ ] معايرة حد الثقة + تحسين تهيئة الصور الممسوحة (DPI / تباين / تدوير تلقائي).
- [ ] استثمار الطبقة المعزولة الحالية لإضافة محرّك بديل دون لمس بقية النظام.

### ب) فرز صريح: مقبولة / مرفوضة + الأسباب
- [ ] قسمان واضحان بعد المعالجة: **المقبولة** (تُعرض حقولها) و**المرفوضة** في قائمة مستقلة مع **سبب الرفض** (صورة غير واضحة / رقم ضريبي مفقود / رقم فاتورة غير مقروء / ضريبة غير متطابقة / الملف لا يحتوي فاتورة واضحة).
- [ ] **عدّ صحيح:** كل فاتورة مُدخلة = عنصر ظاهر للمستخدم (لا تختفي صامتةً بالدمج).

### ج) تجربة المعالجة
- [ ] رسالة تفاعلية واضحة أثناء المعالجة: «جاري معالجة الفواتير الآن، يرجى الانتظار...» بحيث لا يشعر المستخدم أن النظام متوقّف.

### د) ترحيل الفواتير المقبولة إلى الفرع/المحل
- [ ] عرض قائمة الفواتير المقبولة قبل الترحيل.
- [ ] اختيار الفرع/المحل من قائمة منسدلة بكل المحلات المسجّلة.
- [ ] زر **«ترحيل الفواتير»** → إنشاء سجلات مشتريات مربوطة بالفرع المحدّد + إرفاق نسخة كل فاتورة بسجلها.
- [ ] رسالة تأكيد: «تم ترحيل الفواتير المقبولة إلى الفرع المحدد بنجاح».
- [ ] بقاء المرفوضة في شاشة مستقلة لإعادة الرفع/التصحيح لاحقاً.
- ⚠️ **فجوة تقنية:** مسار الـAI الحالي لا يربط سجل المشتريات بالمحل (`shop_id`) — يجب إضافة العمود/الربط في الترحيل.
